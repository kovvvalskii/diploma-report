\section{Реализация интеллектуальной системы мониторинга рисков и обеспечения безопасности путешественников}
\label{sec:development}

В данном разделе подробно описывается процесс программной реализации спроектированной системы, приводятся примеры исходного кода ключевых модулей на языках Python и Java, а также демонстрируются результаты работы пользовательского интерфейса. Разработка велась с применением современных практик непрерывной интеграции, контейнеризации и автоматизированного тестирования, что позволило обеспечить высокую надежность и расширяемость всех компонентов интеллектуальной платформы.

Архитектура разработанного программного комплекса опирается на микросервисный подход и событийно-ориентированное взаимодействие. Выбор данной парадигмы продиктован необходимостью независимого масштабирования вычислительно сложных узлов (таких как анализ текстов нейронными сетями) и узлов, отвечающих за ввод-вывод (сбор данных, отдача картографической информации клиентам). В качестве брокера сообщений, связывающего компоненты системы, выбран RabbitMQ, что обеспечило гарантированную доставку данных между асинхронными сервисами даже в условиях пиковых нагрузок.

\subsection{Разработка модуля сбора и первичной обработки данных}

Модуль Fetcher является входной точкой системы и реализован на языке Python 3.12 с использованием асинхронной парадигмы программирования. Основная сложность реализации данного модуля заключалась в необходимости обработки огромных массивов данных проекта GDELT (Global Database of Events, Language, and Tone), которые публикуются в виде сжатых CSV-файлов каждые 15 минут и могут содержать информацию о тысячах событий одновременно.

Каждое обновление GDELT состоит из трех основных файлов: массива событий (Events), упоминаний в СМИ (Mentions) и графа глобальных знаний (Global Knowledge Graph, GKG). Общий объем распакованных данных за одни сутки может превышать несколько гигабайт, что делает невозможным их обработку традиционными синхронными методами без существенных задержек.

Для обеспечения высокой производительности и минимизации задержек разработан асинхронный клиент на базе библиотеки \textbf{httpx}, позволяющий параллельно загружать архивы событий и упоминаний. Использование \textbf{httpx} вместо стандартной библиотеки \textbf{requests} позволило освободить глобальную блокировку интерпретатора (GIL) на время ожидания ответа от серверов GDELT, за счет чего один процесс может обрабатывать десятки сетевых соединений одновременно.

После загрузки и распаковки в оперативной памяти данные передаются в конвейер обработки, реализованный с применением библиотеки \textbf{pandas}. Данная библиотека, написанная на языке C, обеспечивает векторизованную фильтрацию и очистку данных на низком уровне. В частности, на этапе первичной фильтрации отбрасываются события, индекс Гольдштейна которых не удовлетворяет заданным порогам опасности (например, события с положительным индексом, означающие мирные инициативы, исключаются из обработки как не представляющие угрозы для путешественников).

На рисунке~\ref{fig:code_gdelt_fetcher} представлен фрагмент кода асинхронного сервиса загрузки данных GDELT.

\trmcodefigure[18]{language=Python}{gdelt_fetcher.py}{асинхронного сервиса загрузки данных GDELT}{fig:code_gdelt_fetcher}

Отдельное внимание уделено механизмам отказоустойчивости. При взаимодействии со сторонними API часто возникают сетевые аномалии: обрывы соединений, таймауты или временная недоступность файлов. Для решения этой проблемы в модуле сбора данных реализован паттерн Exponential Backoff (экспоненциальная задержка при повторных попытках). При возникновении ошибки скачивания скрипт повторяет запрос через 2, 4, 8 и 16 секунд, прежде чем пометить файл как недоступный и отправить соответствующее уведомление в систему мониторинга.

Ключевые особенности реализации модуля сбора данных:
\begin{itemize}
    \item применение асинхронных контекстных менеджеров для эффективного управления сетевыми соединениями и ресурсами операционной системы;
    \item использование векторизованных операций библиотеки \textbf{pandas} для быстрой фильтрации записей по пороговым значениям;
    \item реализация отказоустойчивой очереди сообщений на базе RabbitMQ (использование механизмов Publisher Confirms и dead-letter exchanges), гарантирующей доставку идентификаторов событий в аналитический модуль;
    \item внедрение механизмов логирования и мониторинга состояния сетевых запросов;
    \item использование временных хранилищ (in-memory buffers) для минимизации нагрузки на дисковую подсистему при обработке архивов.
\end{itemize}

Данный подход позволил сократить время от момента публикации данных в GDELT до начала их анализа нейронными сетями до 5--10 секунд, что является критически важным для систем экстренного оповещения и мониторинга рисков в реальном времени.

\subsection{Реализация интеллектуального анализа событий}

Модуль интеллектуального анализа представляет собой высокотехнологичный конвейер на базе фреймворка FastAPI, объединяющий классические методы обработки естественного языка и современные генеративные модели. Процесс анализа разделен на этапы извлечения контента, семантической классификации и реферирования, что позволяет гибко настраивать параметры каждой модели в зависимости от типа входных данных и доступных вычислительных ресурсов.

Первым этапом конвейера является извлечение полного текста новостной статьи по URL-адресу, полученному из GDELT. Для этого применяется эвристический парсер на базе библиотеки \textbf{BeautifulSoup4}, который очищает HTML-документ от рекламных баннеров, скриптов, элементов навигации и извлекает исключительно текстовое содержимое статьи. Для обхода блокировок со стороны новостных порталов используются динамические пулы User-Agent и механизмы ротации прокси-серверов.

Для классификации угроз используется энкодерная модель DeBERTa-v3-large, настроенная на задачу классификации без предварительного обучения на целевых классах (Zero-Shot Classification). В отличие от классического подхода с дообучением (Fine-Tuning), Zero-Shot классификация на базе гипотез (NLI - Natural Language Inference) позволяет системе распознавать новые типы инцидентов, динамически подставляя их в промпт. Например, система может классифицировать текст по меткам «террористический акт», «стихийное бедствие», «эпидемия» или «гражданские беспорядки» без необходимости формирования обширной обучающей выборки и затратного переобучения весов нейронной сети.

Особое внимание уделено интеграции с локальной языковой моделью через программный интерфейс Ollama. В качестве базовой генеративной модели выбрана Qwen 2.5 с квантованием до 4 бит (GGUF формат). Это решение обеспечивает полную автономность системы, исключает зависимость от внешних платных провайдеров (таких как OpenAI API или Anthropic) и гарантирует конфиденциальность обрабатываемых данных. Использование квантованных моделей позволяет запускать LLM на потребительском оборудовании с объемом видеопамяти от 8 ГБ, не теряя при этом в качестве генерации аналитических сводок.

На этапе реферирования LLM получает строгий системный промпт (System Prompt), предписывающий выделить из текста суть инцидента, точное местоположение, количество пострадавших (если есть) и потенциальные последствия для транспортной инфраструктуры. Модель генерирует компактную сводку (до 500 символов) и список рекомендуемых действий для путешественников.

На рисунке~\ref{fig:code_intelligence_analyzer} представлен фрагмент кода модуля интеллектуального анализа событий.

\trmcodefigure[20]{language=Python,basicstyle=\ttfamily\scriptsize}{intelligence_analyzer.py}{модуля интеллектуального анализа событий}{fig:code_intelligence_analyzer}

Для поддержки локализации в аналитический конвейер был добавлен этап определения языка исходного материала. С помощью легковесной модели \textbf{fasttext} система за миллисекунды определяет язык оригинальной статьи. После формирования базовой сводки на английском языке, модуль инициирует асинхронные запросы к LLM для подготовки локализованных вариантов названия, описания и рекомендаций для русского, испанского и других поддерживаемых языков. 

Программные решения, внедренные в аналитический модуль:
\begin{itemize}
    \item использование библиотеки \textbf{transformers} для высокоточной семантической классификации;
    \item интеграция с локальным сервером Ollama через REST API для выполнения ресурсоемких задач генерации на выделенных графических мощностях;
    \item ограничение входного контекста для большой языковой модели до 1000 токенов с целью оптимизации времени инференса и предотвращения эффекта потери внимания (lost in the middle);
    \item применение пула воркеров (Worker Pool) и семафоров для контроля параллельной нагрузки на GPU, чтобы избежать ошибок out-of-memory (OOM);
    \item сохранение готовых локализованных версий текстов в кэш-базу данных для моментальной выдачи пользователю без повторного запуска инференса.
\end{itemize}

\subsection{Реализация серверной логики и геоинформационных компонентов}

Ядро системы, отвечающее за бизнес-логику, хранение данных, авторизацию пользователей и выполнение пространственных запросов, реализовано на языке Java 21 с использованием фреймворка Spring Boot 3.3. В качестве основной системы управления базами данных выступает PostgreSQL, выбор которой обусловлен её высокой надежностью, поддержкой JSON-полей и, что самое главное, наличием мощнейшего геоинформационного расширения PostGIS.

Ключевым технологическим преимуществом является использование модуля Hibernate Spatial. Данный модуль интегрируется в слой ORM (Object-Relational Mapping) и позволяет оперировать геометрическими объектами (Point, LineString, Polygon) непосредственно в коде Java-приложения, используя стандартные классы библиотеки JTS (Java Topology Suite). При сохранении сущности в базу данных Hibernate Spatial автоматически транслирует JTS-объекты в формат WKB (Well-Known Binary), понятный базе данных PostGIS.

Для реализации логики геофенсинга~--- поиска пользователей и маршрутов, находящихся в зоне риска,~--- применяются пространственные функции PostGIS, такие как \textbf{ST\_DWithin} (проверка нахождения объектов в пределах заданного радиуса) и \textbf{ST\_Intersects} (проверка пересечения геометрий). Использование пространственных индексов типа GiST (Generalized Search Tree) позволяет выполнять такие запросы за логарифмическое время даже на таблицах с миллионами записей инцидентов. Выполнение геометрических вычислений на стороне СУБД радикально снижает объем данных, передаваемых по сети между базой и приложением, так как в Java-код возвращается только готовый отфильтрованный результат.

Сущности базы данных описывают структуру инцидента и одновременно обеспечивают поддержку многоязычности. Вместо создания множества связанных таблиц для каждого языка, применен современный подход с использованием типа данных \textbf{JSONB}. В одном поле сущности хранится JSON-объект, ключами которого выступают коды языков (en, ru, es), а значениями — локализованные строки. При выборке данных база может извлекать конкретную локаль на лету с помощью функции \textbf{jsonb\_extract\_path\_text}.

Поверх сущностей построен гибкий слой репозиториев на базе JPA Criteria API. Метод \textbf{findEventsForMap} динамически конструирует SQL-запрос, применяя фильтры по bounding box (видимой области экрана пользователя), минимальному уровню угрозы, категориям инцидентов и времени публикации. На рисунке~\ref{fig:code_event_geofeature} представлен фрагмент кода репозитория пространственной выборки событий.

\trmcodefigure[18]{language=Java,basicstyle=\ttfamily\scriptsize}{event_geofeature.java}{репозитория пространственной выборки событий}{fig:code_event_geofeature}

Для отдачи геоданных клиентскому приложению реализован унифицированный контроллер \textbf{GeoServerController}, который агрегирует информацию от разных сервисов-поставщиков (инциденты, сохраненные маршруты, геозоны) и формирует единый валидный GeoJSON-ответ. 

В высоконагруженных сценариях частое обновление карты пользователями может создать чрезмерную нагрузку на СУБД. Для предотвращения этого в серверную часть интегрирован распределенный кэш на базе Redis. Сформированные GeoJSON-ответы кэшируются по ключам, представляющим собой хэш от параметров запроса (координаты углов карты, фильтры). Время жизни кэша (TTL) настроено на 30 секунд, что обеспечивает идеальный баланс между актуальностью данных и производительностью сервера.

\subsection{Реализация системы почтовых уведомлений}

Для гарантированной доставки экстренной информации об угрозах в системе реализована асинхронная рассылка по электронной почте. Этот канал рассматривается как резервный, независимый от работы веб-приложения и мобильных клиентов, а также юридически значимый способ оповещения.

Сервис уведомлений на Java использует библиотеку \textbf{Spring Boot Starter Mail} для взаимодействия с внешним SMTP-сервером. Чтобы длительное ожидание ответа от почтового сервера не блокировало основной поток выполнения бизнес-логики, методы отправки аннотированы как \textbf{@Async}, и их выполнение делегируется пулу потоков \textbf{ThreadPoolTaskExecutor}.

Отдельным сложным этапом разработки стала верстка шаблонов уведомлений. Почтовые клиенты (особенно корпоративные, такие как Microsoft Outlook) используют устаревшие механизмы рендеринга HTML и не поддерживают современные стандарты CSS. Поэтому шаблоны сверстаны с использованием табличной верстки и инлайн-стилей. В проекте используется шаблонизатор \textbf{Thymeleaf}, который позволяет динамически подставлять данные инцидента (название, координаты, уровень угрозы) в предварительно подготовленный HTML-каркас. 

Шаблоны хранятся в каталоге \textbf{resources/templates} проекта по одному файлу на каждую поддерживаемую локаль. В верхней части письма располагается фирменный градиентный заголовок с цветовым индикатором угрозы (желтый, оранжевый, красный). Ниже формируется блок ключевых метаданных события: точный регион, локальное время инцидента и расчетное расстояние от последней известной локации путешественника. Завершает письмо аналитическая сводка от нейросети и нумерованный список рекомендуемых действий. Содержимое почтовых уведомлений о риске безопасности на английском и русском языках представлено на рисунке~\ref{fig:email_notification}.

\subsection{Разработка клиентского интерфейса и ГИС-компонентов}

Фронтенд-приложение построено на базе современной версии React 19 с использованием хуков и функциональных компонентов. Для управления глобальным состоянием приложения (данные о профиле, активные фильтры карты, кэш инцидентов) применяется библиотека Redux Toolkit (RTK). Взаимодействие с API выделено в слой RTK Query, что позволило автоматически кэшировать запросы на клиенте, управлять состояниями загрузки и минимизировать дублирование сетевых вызовов.

Центральным элементом пользовательского интерфейса является интерактивная карта. Для её отрисовки используется библиотека \textbf{react-map-gl}, представляющая собой React-обертку над высокопроизводительным движком \textbf{MapLibre GL JS}. Выбор MapLibre обусловлен его открытой лицензией, поддержкой аппаратного ускорения через WebGL и возможностью работы с векторными тайлами, что выгодно отличает его от растровых решений типа Leaflet.

Для оптимизации рендеринга больших объемов геоданных (десятков тысяч инцидентов) на стороне клиента реализована кластеризация. При отдалении камеры близко расположенные маркеры объединяются в кластеры с отображением количества сгруппированных событий. Это предотвращает падение FPS (частоты кадров) в браузере и делает интерфейс отзывчивым. Применение концепции Data-Driven Styling позволяет прямо в конфигурации слоев карты указывать правила окрашивания маркеров в зависимости от атрибута уровня угрозы (risk\_level) без написания дополнительных скриптов.

Перед началом работы с системой пользователь проходит процедуру аутентификации, реализованную на базе JWT-токенов (JSON Web Token). Окно входа выполнено в минималистичном стиле и поддерживает мгновенное переключение языка. После авторизации токен сохраняется в безопасном хранилище браузера (HttpOnly куки), а пользователь перенаправляется на главный экран.

\subsection{Разработка интерфейса прокладки и анализа маршрута}

Важнейшей аналитической функцией системы является модуль интерактивного планирования маршрута. Пользовательский сценарий реализован в виде пошагового мастера. На первом шаге пользователь добавляет путевые точки кликами по карте или через строку поиска (геокодирование). На втором шаге задается расписание поездки и предполагаемое время нахождения в каждой локации.

Третий шаг — автоматический анализ безопасности — является самым технически сложным на стороне клиента. Модуль вычисляет пересечения отрезков маршрута (сегментов между путевыми точками) с буферными зонами активных инцидентов. Для точного расчета кратчайшего расстояния от точки (инцидента) до отрезка на сфере применяется адаптация математической формулы гаверсинусов (Haversine formula). Формула учитывает кривизну земной поверхности и возвращает результат в километрах.

Пользователь имеет возможность регулировать радиус опасной зоны (буфера) с помощью ползунка в диапазоне от 10 до 500 километров. При изменении радиуса React-компонент мгновенно пересчитывает пересечения и обновляет список предупреждений на панели. На рисунке~\ref{fig:route_step2} представлен второй шаг мастера прокладки маршрута.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=1\linewidth]{picture/route_planning_step2_schedule.png}
    \caption{Второй шаг мастера прокладки маршрута}
    \label{fig:route_step2}
\end{figure}

Каждое предупреждение сопровождается цветовым индикатором и точным расстоянием от линии маршрута до эпицентра события. Если маршрут признан слишком опасным, система предлагает перестроить путь в обход красных зон.

После сохранения все созданные маршруты сериализуются и отправляются на бэкенд, где они сохраняются в базу данных как объекты LineString. Впоследствии фоновый планировщик на сервере будет периодически проверять эти маршруты на пересечение с новыми поступающими инцидентами и, в случае угрозы, отправлять почтовые уведомления.

\subsection{Обеспечение качества, тестирование и CI/CD}

Для поддержания высокого качества исходного кода и стабильности системы в процессе разработки внедрены современные практики CI/CD (Continuous Integration and Continuous Deployment) с использованием платформы GitLab CI. Конвейер автоматизированной сборки запускается при каждом коммите в основную ветку репозитория.

В рамках CI-пайплайна выполняются следующие шаги:
\begin{itemize}
    \item \textbf{линтинг и статический анализ кода} --- проверка Python-кода с помощью \textbf{flake8} и \textbf{black}, проверка Java-кода с помощью \textbf{Checkstyle} и анализ фронтенда через \textbf{ESLint}, это гарантирует соблюдение единого стиля программирования во всей команде;
    \item \textbf{модульное тестирование} --- запуск JUnit тестов для проверки бизнес-логики бэкенда, PyTest для аналитического модуля и Jest для проверки логики Redux-редьюсеров на фронтенде;
    \item \textbf{интеграционное тестирование с Testcontainers} --- на этапе тестирования Spring Boot приложения автоматически поднимаются изолированные Docker-контейнеры с PostgreSQL (PostGIS) и RabbitMQ, что позволяет прогонять тесты взаимодействия с реальной базой данных, не загрязняя локальное окружение разработчика и гарантируя воспроизводимость тестов на любом сервере;
    \item \textbf{сборка Docker-образов} --- при успешном прохождении всех тестов исходный код компилируется, и для каждого микросервиса (Frontend, Backend, NLP-Analyzer, Fetcher) создаются легковесные Docker-образы. Для Java-сервиса применяется многоэтапная сборка (multi-stage build), что позволяет исключить инструменты сборки (Maven) из финального образа, уменьшив его размер в несколько раз;
    \item \textbf{развертывание} --- автоматическое обновление контейнеров на тестовом сервере с использованием утилиты \textbf{Docker Compose}.
\end{itemize}

Внедрение автоматизированного тестирования и контейнеризации позволило сократить время вывода новых функций в продакшен и свело к минимуму риск возникновения регрессионных ошибок при доработке сложной математической логики расчетов геометрии.

\subsection{Вывод по реализации}

В результате дипломного проектирования разработан и успешно протестирован полноценный программный комплекс интеллектуального мониторинга рисков, который полностью отвечает всем заявленным функциональным и нефункциональным требованиям. В ходе реализации решены комплексные инженерные задачи, связанные с обработкой потоковых больших данных, пространственными вычислениями и интеграцией нейросетевых моделей в распределенную микросервисную архитектуру.

Основные результаты этапа разработки:
\begin{itemize}
    \item реализован высокопроизводительный асинхронный модуль сбора данных на Python, способный обрабатывать поток новостей GDELT в реальном времени с поддержкой механизмов отказоустойчивости;
    \item разработан масштабируемый аналитический конвейер, применяющий модель DeBERTa-v3 для Zero-Shot классификации и квантованную LLM Qwen 2.5 для автоматического реферирования и мультиязычного перевода;
    \item создана надежная серверная часть на Java со встроенной поддержкой геопространственных данных Hibernate Spatial и кэшированием Redis;
    \item внедрена база данных PostgreSQL с расширением PostGIS, обеспечивающая сверхбыстрое выполнение пространственных запросов на пересечение маршрутов с использованием GiST-индексов;
    \item разработан современный клиентский SPA-интерфейс на базе React и WebGL, обеспечивающий плавную визуализацию десятков тысяч инцидентов, кластеризацию и интерактивную прокладку безопасных маршрутов;
    \item реализована система асинхронных почтовых рассылок с адаптивными шаблонами, гарантирующая доставку локализованной информации конечному пользователю;
    \item выстроен полный цикл непрерывной интеграции (CI/CD) с применением Docker-контейнеризации и Testcontainers.
\end{itemize}

Спроектированная событийно-ориентированная архитектура на базе брокера очередей RabbitMQ продемонстрировала высокую отказоустойчивость. Все компоненты системы слабо связаны между собой, что обеспечивает широчайший потенциал для дальнейшего горизонтального масштабирования, добавления новых источников данных и внедрения прогностических модулей на базе методов машинного обучения.
