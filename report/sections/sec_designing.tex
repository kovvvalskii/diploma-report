\section{Проектирование системы мониторинга рисков и обеспечения безопасности путешественников}
\label{sec:designing}

В данном разделе приведено описание функциональных требований, архитектурное проектирование компонентов системы и детальное описание структуры данных и алгоритмов взаимодействия. Проектирование выполнено с учетом выявленных в ходе анализа требований к масштабируемости, отказоустойчивости и скорости обработки данных.

\subsection{Функциональные требования и сценарии использования}

Система мониторинга рисков спроектирована как многопользовательская платформа, обеспечивающая непрерывный цикл защиты путешественника. Основные функциональные возможности были разделены на несколько уровней взаимодействия.

Ключевые функциональные требования включают в себя:
\begin{itemize}
    \item \textbf{управление учетными данными:} безопасная регистрация, авторизация через современные протоколы и гибкая настройка персонального профиля безопасности;
    \item \textbf{географическая визуализация:} интерактивное отображение глобальной карты инцидентов с возможностью фильтрации по времени, категориям (например, только медицинские или только транспортные риски) и уровню значимости;
    \item \textbf{динамический анализ маршрутов:} возможность загрузки планируемого пути следования (в формате GPX или через построение на карте) для автоматической проверки на наличие пересечений с активными зонами угроз;
    \item \textbf{проактивная система оповещений:} генерация и доставка мгновенных уведомлений в реальном времени при обнаружении новых инцидентов в радиусе ответственности пользователя;
    \item \textbf{локализация интерфейса и данных:} хранение предпочтительного языка пользователя, отображение интерфейса на выбранном языке и выдача локализованных названий, описаний и почтовых уведомлений;
    \item \textbf{автоматическое реферирование источников:} формирование краткой аналитической сводки события на основе текста исходной статьи или ее машинного перевода на опорный язык;
    \item \textbf{мониторинг инфраструктуры:} инструменты для администратора, позволяющие отслеживать работоспособность микросервисов, нагрузку на большие языковые модели и актуальность базы данных GDELT.
\end{itemize}

Сценарии использования охватывают основные этапы взаимодействия пользователя с системой, начиная от предварительного планирования поездки и заканчивая оперативным реагированием на возникшие угрозы. Для администратора данный сценарий также важен, так как по журналам отправки можно проверить факт попытки информирования пользователя и выявить сбои инфраструктуры отправки почты.

\subsubsection{Модуль сбора данных}

Данный микросервис реализован на языке Python и оптимизирован для выполнения длительных сетевых операций ввода-вывода.
\begin{itemize}
    \item \textbf{планировщик обновлений:} согласно регламенту GDELT 2.0, новые архивы публикуются каждые 15 минут, поэтому модуль использует асинхронный цикл для загрузки и распаковки данных;
    \item \textbf{многоэтапная фильтрация:} для снижения нагрузки на вычислительные ресурсы анализатора модуль сбора данных проводит первичную грубую очистку данных, отсекая события с низким индексом достоверности или находящиеся вне зон интереса системы;
    \item \textbf{обеспечение живучести:} все скачанные сырые данные сохраняются в промежуточную базу данных, что гарантирует возможность повторного анализа при обновлении версий нейросетевых моделей;
    \item \textbf{сигнализация событий:} после успешной подготовки порции данных модуль публикует идентификатор события в RabbitMQ, инициируя дальнейшую цепочку обработки.
\end{itemize}

Архитектурные и ИИ-решения, примененные в модуле:
\begin{itemize}
    \item \textbf{классификация без предварительного обучения на целевых классах:} использование предобученной модели DeBERTa-v3 позволяет классифицировать события по произвольному набору меток без переобучения, это критически важно для адаптации системы к новым видам гибридных угроз;
    \item \textbf{контекстный геопарсинг:} интеграция библиотеки Mordecai позволяет не просто находить названия городов, но и разрешать топонимические конфликты (например, отличать Лондон в Великобритании от Лондона в Канаде) на основе анализа окружающего текста;
    \item \textbf{локализационный контур:} определение языка исходной статьи выполняется до реферирования, после чего модуль переводит текст при отсутствии прямой поддержки языка и формирует краткую сводку события;
    \item \textbf{локальная среда инференса:} использование больших языковых моделей Qwen2.5 или Mistral через Ollama внутри контура системы обеспечивает высокую скорость генерации сводок и исключает передачу конфиденциальных данных на сторонние программные интерфейсы.
\end{itemize}

Одной из ключевых инноваций является расчет динамической зоны влияния инцидента. Если классические системы ставят точку на карте, то анализатор строит полигон-буфер, радиус которого зависит от типа события и его интенсивности, что позволяет более точно уведомлять пользователей.

С учетом ограниченной многоязычности моделей реферирования проектный конвейер строится по принципу опорного языка. Если исходная статья написана на языке, для которого локальная большая языковая модель стабильно выполняет краткое реферирование, текст передается в модель напрямую. Если язык не входит в поддерживаемый набор, модуль сначала использует локальную модель перевода, например NLLB-200 или MarianMT, переводит материал на английский или русский язык, а затем формирует краткую аналитическую сводку. Такой порядок снижает риск некорректной генерации на редких языках и позволяет контролировать качество результата через ограниченный набор проверенных языковых направлений.

\subsubsection{Серверная часть приложения}

Микросервис на Java 21, управляющий бизнес-логикой и интеграцией с конечными пользователями.
\begin{itemize}
    \item \textbf{геопространственная обработка:} использование библиотеки Hibernate Spatial в связке с PostGIS позволяет выполнять операции пространственного пересечения и поиска по расстоянию с высокой скоростью на миллионах записей;
    \item \textbf{алгоритм асинхронного геофенсинга:} при поступлении нового инцидента сервер запускает фоновую задачу по поиску пересечений зоны риска с маршрутами активных пользователей, что не блокирует основной поток обработки;
    \item \textbf{безопасность доступа:} реализация авторизации без хранения состояния на основе JWT-токенов с интеграцией через Spring Security и защитой от перебора паролей.
\end{itemize}

\subsection{Проектирование конвейера реферирования и локализации}

Конвейер подготовки текстового представления события проектируется как последовательность контролируемых этапов, каждый из которых сохраняет промежуточный результат и может быть повторно выполнен при изменении модели или правил обработки. На вход поступает исходный текст статьи, полученный из внешнего источника или из промежуточного хранилища. Далее выполняется определение языка, проверка поддерживаемости языка моделью реферирования, машинный перевод при необходимости, генерация краткой сводки, локализация заголовка и описания, а затем сохранение результата в основную базу данных.

Ключевым проектным решением является отказ от попытки реферировать все языки непосредственно одной универсальной моделью. Несмотря на наличие мультиязычных больших языковых моделей, качество их работы на редких языках и смешанных текстах остается нестабильным. Поэтому в системе вводится понятие опорного языка: если исходный материал написан на английском или русском языке, он может быть передан в локальную модель реферирования напрямую, а для остальных языков сначала выполняется машинный перевод локально развернутой моделью. Такой подход делает качество сводок более предсказуемым и упрощает тестирование, так как проверять необходимо ограниченное число языковых направлений.

Результат реферирования хранится отдельно от полного описания события. Полное описание используется на странице события и в аналитическом интерфейсе, а краткая сводка применяется в списке событий, теме электронного письма, мгновенном уведомлении и компактной карточке на карте. Для этого в сущности \textbf{Event} предусмотрено поле \textbf{summary\_i18n}, организованное по тому же принципу, что \textbf{title\_i18n} и \textbf{description\_i18n}. Если нужная локаль отсутствует, серверная часть использует резервную версию на опорном языке, но сам факт отсутствия перевода фиксируется в журнале обработки для последующей повторной генерации.

Для предотвращения искажения смысла в сводке проектируется дополнительная проверка ключевых сущностей. После генерации краткого текста система сопоставляет найденные в нем географические названия, даты, числовые значения и тип угрозы с результатами предыдущих этапов анализа. Если сводка не содержит основной локации или противоречит классификации события, результат помечается как требующий повторной обработки с более строгим промптом или с использованием другой модели. Такой механизм особенно важен для уведомлений, поскольку пользователь принимает решение на основе одного-двух предложений.

С точки зрения производительности реферирование выполняется асинхронно и не блокирует прием новых сообщений из очереди. Для критических событий сначала может быть сохранена минимальная карточка с категорией, геометрией и ссылкой на источник, а затем, после завершения перевода и реферирования, запись обновляется локализованными текстовыми полями. Это позволяет не задерживать геопространственную обработку и при этом постепенно повышать качество пользовательского представления события.

\subsubsection{Промежуточное хранилище}

Данный слой предназначен для кратковременного хранения необработанной информации из GDELT. Основная цель его выделения — разделение процессов интенсивной загрузки данных и аналитического чтения.
\begin{itemize}
    \item \textbf{таблица gdelt\_events:} содержит плоские данные из CSV-файлов, включая уникальные идентификаторы GDELT и первичные метки событий;
    \item \textbf{таблица raw\_articles:} кэширует полные тексты новостей, что позволяет избежать повторного обращения к внешним сайтам при необходимости повторной обработки инцидента.
\end{itemize}

\subsubsection{Основное хранилище}

Спроектировано для обеспечения максимальной производительности ГИС-запросов и аналитики. Ключевые таблицы оптимизированы с помощью пространственных индексов GiST.
\begin{itemize}
    \item сущность \textbf{Event} представляет инцидент, хранящий геометрию зоны риска, локализованные текстовые поля и рассчитанный уровень угрозы;
    \item сущность \textbf{User} представляет профиль пользователя, хранящий электронную почту, предпочтительную локаль и настройки приватности, включая флаги разрешения на отслеживание местоположения;
    \item сущность \textbf{UserLocation} представляет текущее положение пользователя, необходимое для оперативной проверки попадания в активную зону риска;
    \item сущность \textbf{Route} представляет маршрут пользователя, хранящий геометрию пути и временные метки прохождения ключевых точек;
    \item сущность \textbf{Mention} представляет упоминание события во внешнем источнике, содержащее ссылку, язык текста и дату публикации материала;
    \item сущность \textbf{Source} представляет справочник источников данных, позволяющий связывать событие с конкретным новостным доменом или агрегатором;
    \item сущность \textbf{EventType} представляет справочник типов событий, определяющий базовую категорию угрозы и радиус зоны влияния по умолчанию.
\end{itemize}

Для хранения локализованного содержания событий выбран тип JSONB в PostgreSQL. В отличие от отдельной таблицы переводов, JSONB позволяет хранить компактный объект, в котором ключами выступают языковые коды, а значениями являются локализованные строки, непосредственно в записи события, что уменьшает количество соединений при чтении карты и страниц инцидентов. При этом PostgreSQL сохраняет возможность индексирования ключей JSONB, проверки наличия конкретной локали и частичного обновления переводов без пересоздания всей строки.

В сущности \textbf{Event} предусмотрены поля \textbf{title\_i18n}, \textbf{description\_i18n} и \textbf{summary\_i18n}, где ключами выступают ISO-коды языков, а значениями являются локализованные версии названия, описания и краткой сводки. Дополнительно хранится \textbf{source\_language}, фиксирующий язык исходной статьи, чтобы аналитический модуль мог оценивать качество перевода, а пользователь или администратор могли вернуться к первичному тексту при спорных случаях. Для пользовательского профиля в сущности \textbf{User} вводится поле \textbf{preferred\_locale}, которое используется серверной частью при выборе языка ответа программного интерфейса и при генерации шаблонов почтовых уведомлений.

Выбор JSONB является компромиссом между гибкостью и нормализацией. На раннем этапе проекта список поддерживаемых языков может расширяться, поэтому жесткая схема с колонками \textbf{title\_ru}, \textbf{title\_en} и аналогичными полями быстро приведет к миграциям при добавлении новых локалей. Отдельная таблица переводов удобна для очень больших словарей, но в данном случае каждое событие содержит ограниченное количество текстовых фрагментов, которые почти всегда читаются вместе с основной записью, поэтому JSONB снижает сложность запросов и хорошо соответствует модели выдачи данных для карты, страницы события и уведомлений.

\subsection{Проектирование программных интерфейсов}

Для взаимодействия клиентской и серверной частей используется программный интерфейс в стиле REST, соответствующий спецификации OpenAPI 3.0, что обеспечивает простоту интеграции и возможность автоматической генерации клиентских библиотек.

Основные эндпоинты системы спроектированы по принципу минимальной достаточности данных:
\begin{itemize}
    \item \textbf{GET /api/v1/events/map:} выдает массив событий в формате GeoJSON, оптимизированном для отрисовки на мобильных устройствах;
    \item \textbf{GET /api/v1/events/\{id\}:} возвращает детальную карточку события с выбором локализованных полей по параметру \textbf{locale} или по языку профиля пользователя;
    \item \textbf{POST /api/v1/user/location:} обеспечивает высокочастотное обновление координат пользователя с сохранением истории для последующего анализа безопасности;
    \item \textbf{POST /api/v1/trips/analyze:} сложный запрос, принимающий геометрию маршрута и возвращающий детализированный отчет о потенциальных рисках на пути следования;
    \item \textbf{WS /api/v1/notifications:} использование протокола WebSocket для мгновенной доставки уведомлений без необходимости постоянного опроса сервера.
\end{itemize}

Для обеспечения высокой надежности взаимодействия между сервисами внутри контура используется протокол gRPC, который позволяет передавать бинарные данные с минимальными накладными расходами на сериализацию, что особенно важно при передаче больших текстов статей.

\subsection{Выводы по проектированию}

В процессе проектирования была разработана современная архитектура интеллектуальной системы, объединяющая микросервисный подход и передовые технологии искусственного интеллекта. Использование событийно-ориентированной модели в сочетании с Docker-контейнеризацией обеспечивает необходимый уровень масштабируемости и гибкости. Проектирование базы данных на основе PostGIS и внедрение асинхронного геофенсинга позволяет системе обрабатывать глобальные потоки данных в режиме реального времени. Сформированные функциональные требования и спроектированные программные интерфейсы закладывают основу для успешной реализации программного комплекса, способного обеспечить новый уровень безопасности для путешественников.
