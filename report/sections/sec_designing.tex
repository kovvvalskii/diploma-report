\section{Проектирование системы мониторинга рисков и обеспечения безопасности путешественников}
\label{sec:designing}

В данном разделе приведено описание функциональных требований, архитектурное проектирование компонентов системы и детальное описание структуры данных и алгоритмов взаимодействия. Проектирование выполнено с учетом выявленных в ходе анализа требований к масштабируемости, отказоустойчивости и скорости обработки данных.

\subsection{Функциональные требования и сценарии использования}

Система мониторинга рисков спроектирована как многопользовательская платформа, обеспечивающая непрерывный цикл защиты путешественника. Основные функциональные возможности были разделены на несколько уровней взаимодействия.

Ключевые функциональные требования включают в себя:
\begin{itemize}
    \item \textbf{управление учетными данными:} безопасная регистрация, авторизация через современные протоколы и гибкая настройка персонального профиля безопасности;
    \item \textbf{географическая визуализация:} интерактивное отображение глобальной карты инцидентов с возможностью фильтрации по времени, категориям (например, только медицинские или только транспортные риски) и уровню значимости;
    \item \textbf{динамический анализ маршрутов:} возможность загрузки планируемого пути следования (в формате GPX или через построение на карте) для автоматической проверки на наличие пересечений с активными зонами угроз;
    \item \textbf{проактивная система оповещений:} генерация и доставка Push-уведомлений в реальном времени при обнаружении новых инцидентов в радиусе ответственности пользователя;
    \item \textbf{мониторинг инфраструктуры:} инструменты для администратора, позволяющие отслеживать работоспособность микросервисов, нагрузку на LLM-модели и актуальность базы данных GDELT.
\end{itemize}

\subsubsection{Модуль сбора данных (Fetcher)}

Данный микросервис реализован на языке Python и оптимизирован для выполнения длительных сетевых операций ввода-вывода.
\begin{itemize}
    \item \textbf{планировщик обновлений:} согласно регламенту GDELT 2.0, новые архивы публикуются каждые 15 минут, поэтому модуль использует асинхронный цикл для загрузки и распаковки данных;
    \item \textbf{многоэтапная фильтрация:} для снижения нагрузки на вычислительные ресурсы анализатора, Fetcher проводит первичную грубую очистку данных, отсекая события с низким индексом достоверности или находящиеся вне зон интереса системы;
    \item \textbf{обеспечение живучести:} все скачанные сырые данные сохраняются в промежуточную базу данных (Staging DB), что гарантирует возможность повторного анализа при обновлении версий нейросетевых моделей;
    \item \textbf{сигнализация событий:} после успешной подготовки порции данных модуль публикует идентификатор события в RabbitMQ, инициируя дальнейшую цепочку обработки.
\end{itemize}

\subsubsection{Модуль интеллектуального анализа событий (Analyzer)}

Это наиболее сложный компонент системы, реализующий современный NLP-конвейер. В отличие от классических систем, использующих жесткие правила, Analyzer опирается на семантическое понимание текста. Алгоритм работы модуля детально представлен на рисунке \ref{fig:analyzer_flow}.

\begin{figure}[h]
    \centering
    \includegraphics[width=1.0\textwidth]{picture/diagramm/analyzer_flow.png}
    \caption{Алгоритм обработки события в модуле Analyzer}
    \label{fig:analyzer_flow}
\end{figure}

Архитектурные и ИИ-решения, примененные в модуле:
\begin{itemize}
    \item \textbf{Zero-shot классификация (DeBERTa-v3):} использование предобученной модели позволяет классифицировать события по произвольному набору меток без переобучения, это критически важно для адаптации системы к новым видам гибридных угроз;
    \item \textbf{контекстный геопарсинг:} интеграция библиотеки Mordecai позволяет не просто находить названия городов, но и разрешать топонимические конфликты (например, отличать Лондон в Великобритании от Лондона в Канаде) на основе анализа окружающего текста;
    \item \textbf{локальная инференс-среда (Ollama):} использование LLM Qwen2.5 или Mistral внутри контура системы обеспечивает высокую скорость генерации суммаризаций и исключает передачу конфиденциальных данных на сторонние API-сервисы.
\end{itemize}

Одной из ключевых инноваций является расчет динамической «зоны влияния» (Impact Area) инцидента. Если классические системы ставят точку на карте, то наш анализатор строит полигон (Buffer), радиус которого зависит от типа события и его интенсивности, что позволяет более точно уведомлять пользователей.

\subsubsection{Серверная часть приложения (Core Service)}

Микросервис на Java 21, управляющий бизнес-логикой и интеграцией с конечными пользователями.
\begin{itemize}
    \item \textbf{геопространственная обработка:} использование библиотеки Hibernate Spatial в связке с PostGIS позволяет выполнять операции `ST_Intersects` и `ST_DWithin` с высокой скоростью на миллионах записей;
    \item \textbf{алгоритм асинхронного геофенсинга:} при поступлении нового инцидента сервер запускает фоновую задачу по поиску пересечений зоны риска с маршрутами активных пользователей, что не блокирует основной поток обработки;
    \item \textbf{безопасность доступа:} реализация Stateless-авторизации на основе JWT-токенов с интеграцией через Spring Security и защитой от перебора паролей.
\end{itemize}

\subsubsection{Промежуточное хранилище (Staging Layer)}

Данный слой предназначен для кратковременного хранения необработанной информации из GDELT. Основная цель его выделения — разделение процессов интенсивной записи (Ingestion) и аналитического чтения.
\begin{itemize}
    \item \textbf{таблица gdelt\_events:} содержит плоские данные из CSV-файлов, включая уникальные идентификаторы GDELT и первичные метки событий;
    \item \textbf{таблица raw\_articles:} кэширует полные тексты новостей, что позволяет избежать повторного обращения к внешним сайтам при необходимости повторной обработки инцидента.
\end{itemize}

\subsubsection{Основное хранилище (Production Layer)}

Спроектировано для обеспечения максимальной производительности ГИС-запросов и аналитики. Ключевые таблицы оптимизированы с помощью пространственных индексов GiST.
\begin{itemize}
    \item \textbf{entity Event:} основная сущность инцидента, хранящая геометрию зоны риска (тип Geometry), семантическое описание и рассчитанный уровень угрозы;
    \item \textbf{entity User:} хранит профили пользователей и настройки их приватности, включая флаги разрешения на отслеживание местоположения;
    \item \textbf{entity Route:} представляет собой сложный объект LineString, хранящий геометрию пути пользователя и временные метки прохождения ключевых точек;
    \item \textbf{таблица notification\_log:} необходима для ведения аудита и предотвращения повторных уведомлений об одном и том же событии.
\end{itemize}

\subsection{Проектирование программных интерфейсов}

Для взаимодействия фронтенда и бэкенда используется RESTful API, соответствующий спецификации OpenAPI 3.0, что обеспечивает простоту интеграции и возможность автоматической генерации клиентских библиотек.

Основные эндпоинты системы спроектированы по принципу минимальной достаточности данных:
\begin{itemize}
    \item \textbf{GET /api/v1/events/map:} выдает массив событий в формате GeoJSON, оптимизированном для отрисовки на мобильных устройствах;
    \item \textbf{POST /api/v1/user/location:} обеспечивает высокочастотное обновление координат пользователя с сохранением истории для последующего анализа безопасности;
    \item \textbf{POST /api/v1/trips/analyze:} сложный запрос, принимающий геометрию маршрута и возвращающий детализированный отчет о потенциальных рисках на пути следования;
    \item \textbf{WS /api/v1/notifications:} использование протокола WebSocket для мгновенной доставки уведомлений без необходимости постоянного опроса сервера (Polling).
\end{itemize}

Для обеспечения высокой надежности взаимодействия между сервисами внутри контура используется протокол gRPC, который позволяет передавать бинарные данные с минимальными накладными расходами на сериализацию, что особенно важно при передаче больших текстов статей.

\subsection{Выводы по проектированию}

В процессе проектирования была разработана современная архитектура интеллектуальной системы, объединяющая микросервисный подход и передовые технологии искусственного интеллекта. Использование событийно-ориентированной модели в сочетании с Docker-контейнеризацией обеспечивает необходимый уровень масштабируемости и гибкости. Проектирование базы данных на основе PostGIS и внедрение асинхронного геофенсинга позволяет системе обрабатывать глобальные потоки данных в режиме реального времени. Сформированные функциональные требования и спроектированные API закладывают основу для успешной реализации программного комплекса, способного обеспечить новый уровень безопасности для путешественников.
